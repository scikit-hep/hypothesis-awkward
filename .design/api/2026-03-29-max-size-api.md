# API Design: `max_size` and `max_leaf_size` Parameters

- **Date:** 2026-03-29
- **Status:** Proposed
- **Author:** Claude (with developer collaboration)

## Overview

This document describes the API for renaming the current `max_size` parameter to
`max_leaf_size` and introducing a new `max_size` that bounds the total
`content_size()` of generated content.

See
[total-size-budget-research](../research/2026-03-29-total-size-budget-research.md)
for background on the problem and implementation options.

## Background

The current `max_size` parameter in `contents()` and `arrays()` only limits leaf
elements (data scalars, strings, bytestrings). Index buffers generated by
wrapper types (`ListOffsetArray` offsets, `ListArray` starts/stops, `UnionArray`
tags/index) are unbounded. This means a tree with `max_size=10` leaf elements
can generate arbitrarily large index buffers.

## Design Goals

1. **Bound total content size**: A single parameter (`max_size`) that limits the
   total number of scalars stored in the content tree.
2. **Optional leaf-level control**: A separate parameter (`max_leaf_size`) for
   users who want to cap the total number of leaf elements.
3. **Backward-compatible defaults**: `max_leaf_size=None` means existing
   behavior is unchanged when only `max_size` is provided.

## `content_size()`

`max_size` is defined as the upper bound on `content_size(result)`.
`content_size` is a function over the content tree:

In the formulas below, `n = len(result)`.

```text
content_size(NumpyArray)       = len(data)
content_size(EmptyArray)       = 0
content_size(RegularArray)     = 1 + content_size(child)
content_size(RecordArray)      = num_fields + sum(content_size(c) for c in children)
content_size(ListOffsetArray)  = (n + 1) + content_size(child)
content_size(ListArray)        = 2n + content_size(child)
content_size(UnionArray)       = 2n + sum(content_size(c) for c in children)
```

- **`RegularArray`**: The `size` parameter counts as 1, regardless of its value.
  This prevents arbitrarily deep nesting of `RegularArray` within a finite
  budget.
- **`RecordArray`**: Each field name counts as 1, regardless of character
  length. Field names are Python strings, not Awkward buffers. When `fields` is
  `None` (tuple records), `num_fields` is 0.
- **String/bytestring**: These are `ListOffsetArray` wrapping
  `NumpyArray(uint8)`. No special formula — the `ListOffsetArray` and
  `NumpyArray` rules apply directly.
- **`EmptyArray`**: Costs 0. It is a leaf and cannot nest.

## Parameters

### `max_size`

Upper bound on `content_size(result)`.

- **Type:** `int`
- **Default:** `50`
- **Rationale:** Large enough for a few nesting levels with index/offset
  overhead.

### `max_leaf_size`

Upper bound on the total number of leaf elements across the entire content tree.
This is a global cap — leaf elements in all branches (including all children of
`RecordArray` and `UnionArray`) count toward the same budget. Replaces the
current `max_size` parameter with a default of `None` instead of 10.

- **Type:** `int | None`
- **Default:** `None`
- **Rationale:** Total leaf size is bounded by `max_size` by default. When
  provided, the total number of leaf elements across the tree is additionally
  capped at `max_leaf_size`.

### Leaf strategies

`numpy_array_contents`, `string_contents`, `bytestring_contents`, and
`leaf_contents` keep `max_size` with its current meaning ("max elements"). The
rename to `max_leaf_size` only affects `contents()` and `arrays()`.

When `contents()` calls a leaf strategy, it passes
`max_size=min(remaining_budget, max_leaf_size)` (treating `None` as unbounded),
where `remaining_budget` is the portion of `max_size` left after deducting
wrapper overhead from ancestor nodes.

### `regular_array_contents()`

`regular_array_contents()` is a wrapper strategy, not a leaf. Its existing
`max_size` parameter means "max group size" (the `RegularArray.size` value), not
"max elements." This is a naming collision unrelated to the
`max_size`/`max_leaf_size` rename. See the research doc's
"regular_array_contents naming collision" section.

### `content_lists()`

`content_lists()` keeps `max_total_size` as its parameter name. Its semantics
change: it tracks `content_size()` instead of `_leaf_size()` when measuring each
child's consumed budget. The function is defined as:

```python
def _content_size(c: Content) -> int:
    '''Count total scalars stored in a content tree.'''
    match c:
        case NumpyArray():
            return len(c.data)
        case EmptyArray():
            return 0
        case RegularArray():
            return 1 + _content_size(c.content)
        case RecordArray():
            n_fields = len(c.fields) if c.fields is not None else 0
            return n_fields + sum(_content_size(child) for child in c.contents)
        case ListOffsetArray():
            return len(c.offsets.data) + _content_size(c.content)
        case ListArray():
            return len(c.starts.data) + len(c.stops.data) + _content_size(c.content)
        case UnionArray():
            return (len(c.tags.data) + len(c.index.data)
                    + sum(_content_size(child) for child in c.contents))
```

For `UnionArray`, the tags/index overhead (`2n`) is included in the union's
`content_size()`. However, `content_lists()` measures budget consumption _per
child_, and the tags/index arrays are only created after the children are drawn.
How to account for this overhead during budget distribution is an implementation
question — see the research doc's "Resolution options" and "Multi-child
wrappers" sections.

## Interaction with Existing Parameters

### `max_size` and `max_leaf_size`

Both constraints hold simultaneously. `max_size` bounds the total content tree
size. `max_leaf_size`, when provided, additionally caps the total leaf elements.

| Scenario           | `max_size` | `max_leaf_size` | Effect                                              |
| ------------------ | ---------- | --------------- | --------------------------------------------------- |
| Default            | 50         | `None`          | Total content_size <= 50, no leaf cap               |
| With leaf cap      | 50         | 10              | Total content_size <= 50, total leaf elements <= 10 |
| Tight total budget | 20         | `None`          | Total content_size <= 20, leaves bounded by budget  |
| Both tight         | 20         | 5               | Total content_size <= 20, total leaf elements <= 5  |

### `max_size` and `max_length`

These are orthogonal. `max_length` constrains the immediate `len()` of a content
node at one level. `max_size` constrains the total `content_size()` across the
entire tree.

See [max-length-api](2026-02-23-max-length-api.md) for `max_length` semantics.

### `max_size` and `max_depth`

These are complementary. `max_depth` limits nesting depth. `max_size` limits
total scalars. A deep tree with small leaves can stay within `max_size`; a
shallow tree with large leaves can exhaust it.

## Design Decisions

### 1. `max_leaf_size` Defaults to `None`

**Decision:** `max_leaf_size` defaults to `None`, not 10.

**Rationale:** The new `max_size` provides the primary size constraint. Having
both default to finite values would double-constrain generation unnecessarily.
`None` means "no leaf cap beyond what `max_size` allows."

### 2. `max_size` Defaults to 50

**Decision:** `max_size` defaults to 50, not 10.

**Rationale:** Under the new definition, `content_size` includes offsets,
indices, tags, and metadata. A value of 10 would be exhausted by a single
`ListOffsetArray` with a few elements. 50 gives room for a few nesting levels.

### 3. Leaf Strategies Keep `max_size`

**Decision:** Leaf strategies (`numpy_array_contents`, `string_contents`,
`bytestring_contents`, `leaf_contents`) keep their `max_size` parameter with
current semantics.

**Rationale:** At the leaf level, `max_size` means "max elements" — the same
dimension as `content_size` for a flat array. Renaming would add complexity
without benefit. The `contents()` tree builder translates between the two
namespaces.

### 4. `content_lists()` Keeps `max_total_size`

**Decision:** `content_lists()` keeps `max_total_size` as its parameter name.
Only the tracking function changes from `_leaf_size()` to `_content_size()`.

**Rationale:** `max_total_size` already means "total budget across children."
The semantic change (counting all scalars instead of just leaves) aligns the
name's meaning with its new behavior.

## Implementation Note

This document specifies the API (parameters, signatures, `content_size()`
definition). The strategy for **enforcing** the `max_size` budget top-down
during tree generation is an implementation question explored in the research
doc's "Resolution options" section (Options A–E). The choice of enforcement
strategy does not affect the public API.
