# API Design: Option Content Strategies

- **Date:** 2026-03-27
- **Status:** Proposed
- **Author:** Claude (with developer collaboration)

## Overview

This document describes the API for the per-type option content strategies and
the combined `option_contents()` selector. A separate document covers
integration into `contents()` and `arrays()`.

See [option-types-research](../research/2026-03-27-option-types-research.md) for
background on Awkward Array's option-type Content classes.

## Background

Awkward Array has four option-type Content classes that represent missing
(nullable) data:

- **`IndexedOptionArray`** — Index-based; negative index = None
- **`ByteMaskedArray`** — Byte mask (int8); `valid_when` flag
- **`BitMaskedArray`** — Bit-packed mask (uint8); `valid_when` + `lsb_order`
- **`UnmaskedArray`** — No mask; option type with no actual nulls

All four share a common constraint: they cannot directly wrap another option
type, an indexed type, or a union type.

## Design Goals

1. **One strategy per option type**: Each option-type Content class gets its own
   strategy, following the pattern of `numpy_array_contents()`,
   `regular_array_contents()`, etc.
2. **Combined selector**: `option_contents()` picks among the per-type
   strategies via `st.one_of`, following the `leaf_contents()` pattern.
3. **Composable**: Each strategy accepts `content` as
   `st.SearchStrategy[Content] | Content | None`, enabling both standalone use
   and composition via `contents()`.
4. **Valid-by-construction**: Generated option nodes always wrap non-option,
   non-indexed, non-union content. The caller is responsible for providing valid
   inner content.

## API

### `indexed_option_array_contents()`

Wraps content in an `IndexedOptionArray` with a generated index.

```python
@st.composite
def indexed_option_array_contents(
    draw: st.DrawFn,
    content: st.SearchStrategy[Content] | Content | None = None,
    *,
    max_length: int | None = None,
) -> IndexedOptionArray:
```

#### Parameters

- **`content`** — Inner content to wrap. Can be a strategy for Content, a
  concrete Content instance, or `None` to draw from `contents()`.
- **`max_length`** — Upper bound on `len(result)`, i.e., the length of the
  index. No constraint when `None`.

#### Behavior

1. Resolve `content` (draw if strategy, use as-is if concrete, draw from
   `contents()` if `None`).
2. Draw an index length, respecting `max_length`.
3. For each index position, draw either `-1` (missing) or a valid index into
   content (`0 <= i < len(content)`). When `len(content) == 0`, all entries are
   `-1`.
4. Draw index dtype from `{int32, int64}`.
5. Construct `IndexedOptionArray(index, content)`.

### `byte_masked_array_contents()`

Wraps content in a `ByteMaskedArray` with a generated byte mask.

```python
@st.composite
def byte_masked_array_contents(
    draw: st.DrawFn,
    content: st.SearchStrategy[Content] | Content | None = None,
    *,
    max_length: int | None = None,
) -> ByteMaskedArray:
```

#### Parameters

- **`content`** — Inner content to wrap. Same resolution as above.
- **`max_length`** — Upper bound on `len(result)`, i.e., `len(mask)`. No
  constraint when `None`.

#### Behavior

1. Resolve `content`.
2. The mask length equals `len(content)` (or capped by `max_length`).
3. Draw a boolean mask of that length, then convert to int8.
4. Draw `valid_when` from `{True, False}`.
5. Construct `ByteMaskedArray(mask, content, valid_when)`.

### `bit_masked_array_contents()`

Wraps content in a `BitMaskedArray` with a generated bit-packed mask.

```python
@st.composite
def bit_masked_array_contents(
    draw: st.DrawFn,
    content: st.SearchStrategy[Content] | Content | None = None,
    *,
    max_length: int | None = None,
) -> BitMaskedArray:
```

#### Parameters

- **`content`** — Inner content to wrap. Same resolution as above.
- **`max_length`** — Upper bound on `len(result)`, i.e., the `length` parameter.
  No constraint when `None`.

#### Behavior

1. Resolve `content`.
2. The logical length equals `len(content)` (or capped by `max_length`).
3. Compute mask byte count: `ceil(length / 8)`.
4. Draw mask bytes as uint8.
5. Draw `valid_when` from `{True, False}`.
6. Draw `lsb_order` from `{True, False}`.
7. Construct `BitMaskedArray(mask, content, valid_when, length, lsb_order)`.

### `unmasked_array_contents()`

Wraps content in an `UnmaskedArray` (option type with no actual nulls).

```python
@st.composite
def unmasked_array_contents(
    draw: st.DrawFn,
    content: st.SearchStrategy[Content] | Content | None = None,
    *,
    max_length: int | None = None,
) -> UnmaskedArray:
```

#### Parameters

- **`content`** — Inner content to wrap. Same resolution as above.
- **`max_length`** — Upper bound on `len(result)`. Since `UnmaskedArray`
  delegates length to its content, this constrains the content's length when
  drawing from `contents()`. No constraint when `None`.

#### Behavior

1. Resolve `content`. When drawing from `contents()`, pass `max_length` through.
2. Construct `UnmaskedArray(content)`.

### `option_contents()`

Combined selector that picks among the per-type option strategies via
`st.one_of`, following the `leaf_contents()` pattern.

```python
def option_contents(
    content: st.SearchStrategy[Content] | Content | None = None,
    *,
    max_size: int = 10,
    max_length: int | None = None,
    allow_indexed_option: bool = True,
    allow_byte_masked: bool = True,
    allow_bit_masked: bool = True,
    allow_unmasked: bool = True,
) -> st.SearchStrategy[Content]:
```

#### Parameters

- **`content`** — Forwarded to each per-type strategy.
- **`max_size`** — Forwarded to the inner content strategy when `content` is
  `None`. Allows `option_contents` to conform to the `_StContent` protocol used
  by `content_lists()`.
- **`max_length`** — Forwarded to each per-type strategy.
- **`allow_indexed_option`** — Include `indexed_option_array_contents()` if
  `True`.
- **`allow_byte_masked`** — Include `byte_masked_array_contents()` if `True`.
- **`allow_bit_masked`** — Include `bit_masked_array_contents()` if `True`.
- **`allow_unmasked`** — Include `unmasked_array_contents()` if `True`.

#### Behavior

Builds a list of per-type strategies based on the `allow_*` flags, then returns
`st.one_of(options)`. Raises `ValueError` if all flags are `False`.

## Design Decisions

### 1. `content` as First Positional Parameter

**Decision:** Each per-type strategy takes `content` as the first positional
parameter (Group A in the positional-keyword convention), matching
`list_offset_array_contents(content, ...)`,
`regular_array_contents(content, ...)`, etc.

**Rationale:** Option strategies are wrappers — "what to wrap" is the primary
input, just like list and regular array strategies.

### 2. `max_length` on Per-Type Strategies

**Decision:** Each per-type strategy accepts `max_length` as a keyword-only
parameter.

**Rationale:** Consistent with all other wrapper strategies
(`list_offset_array_contents`, `regular_array_contents`,
`record_array_contents`, `union_array_contents`). At the root level,
`contents()` passes `max_length` to the chosen wrapper. For
`IndexedOptionArray`, `max_length` caps `len(index)`. For `ByteMaskedArray`, it
caps `len(mask)`. For `BitMaskedArray`, it caps the `length` parameter. For
`UnmaskedArray`, it constrains the content's length.

### 3. Mask Length Equals Content Length

**Decision:** For `ByteMaskedArray` and `BitMaskedArray`, the mask length
matches `len(content)` (unless capped by `max_length`). No inaccessible content
is generated.

**Rationale:** Simpler generation. Awkward allows content to be longer than the
mask, but the extra elements are never accessible. Generating inaccessible
content adds complexity without improving test coverage. This can be revisited
later if needed.

### 4. IndexedOptionArray Uses Compact Indexing

**Decision:** For the initial implementation, `indexed_option_array_contents()`
generates indices where valid entries use compact sequential indexing
(`index[i] = i` for valid entries, `-1` for missing). The index length equals
`len(content)`.

**Rationale:** Compact indexing avoids needing content larger than the array.
Non-compact indexing (reordering, deduplication) exercises `IndexedArray`
semantics rather than option semantics. It can be added later via an
`allow_non_compact` parameter.

**Alternative considered:** Drawing arbitrary valid indices into content. This
would require the content to be potentially larger than the array, complicating
the size budget. Deferred for now.

### 5. `max_size` Passed Through to Content

**Decision:** `option_contents()` accepts a `max_size` parameter and forwards it
to the inner content strategy. This is needed so `option_contents` can conform
to the `_StContent` protocol (callable accepting `max_size`) used by
`content_lists()`.

**Rationale:** Option wrappers add no scalars themselves, but when used inside
`content_lists()` (e.g., for UnionArray children), they receive the remaining
scalar budget via `max_size` and must forward it to the inner content strategy.

```text
src/hypothesis_awkward/strategies/contents/
├── ...existing files...
├── indexed_option_array.py    # indexed_option_array_contents()
├── byte_masked_array.py       # byte_masked_array_contents()

```

```python

    'byte_masked_array_contents',
    'bit_masked_array_contents',
    'unmasked_array_contents',
    'option_contents',

```

## Relationship Diagram

```text
Per-type option strategies:
  indexed_option_array_contents()  ──┐
  byte_masked_array_contents()     ──┤
  bit_masked_array_contents()      ──┼──>  option_contents()  ──>  contents()
  unmasked_array_contents()        ──┘
```

Each per-type strategy wraps non-option content:

```text
indexed_option_array_contents(content=NumpyArray(...)):

  IndexedOptionArray(index=[-1, 0, 1, -1, 2], content=NumpyArray)
```

## Open Questions

1. **Should `IndexedOptionArray` support non-compact indexing?** Currently,
   valid indices are sequential. Non-compact indexing (reordering, skipping)
   exercises different code paths. Could be added via an `allow_non_compact`
   parameter.

2. **Should mask/index length differ from content length?** Currently, mask and
   content lengths match. Allowing content to be longer would exercise Awkward's
   handling of inaccessible elements.
