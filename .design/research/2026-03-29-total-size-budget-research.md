# Research: Total Size Budget (`max_size` / `max_leaf_size`)

- **Date:** 2026-03-29
- **Status:** Research
- **Author:** Claude (with developer collaboration)

## Problem

The current `max_size` parameter in `contents()` and `arrays()` only limits the
number of **leaf elements** (data scalars, strings, bytestrings). It does not
account for **index buffers** generated by wrapper content types:

- `ListOffsetArray`: `n + 1` int64 offsets
- `ListArray`: `2n` int64 values (starts + stops)
- `UnionArray`: `n` int8 tags + `n` int64 indices

These index/offset buffers can be arbitrarily large relative to the leaf data
they describe. A tree with `max_size=10` leaf elements could generate hundreds
of offset/index values.

Additionally, `max_size` is passed unchanged from parent to child for
single-child wrappers (`regular`, `list_offset`, `list`). The budget is only
distributed across children for `record` and `union` (via `content_lists()`).

## Goal

Rename `max_size` to `max_leaf_size` (defaulting to `None` instead of 10), and
introduce a new `max_size` as the upper bound on `content_size()` — the total
number of scalars stored in the content tree.

## Current State

### Where `max_size` appears

| Location                             | Current meaning                                                         |
| ------------------------------------ | ----------------------------------------------------------------------- |
| `arrays(max_size=10)`                | Forwards to `contents()`                                                |
| `contents(max_size=10)`              | Leaf element budget; distributed via `content_lists()` for record/union |
| `numpy_array_contents(max_size=10)`  | Max elements in the NumpyArray                                          |
| `string_contents(max_size=10)`       | Max number of strings                                                   |
| `bytestring_contents(max_size=10)`   | Max number of bytestrings                                               |
| `regular_array_contents(max_size=5)` | Max group size (different semantics!)                                   |
| `content_lists(max_total_size=10)`   | Total leaf budget across children                                       |
| `leaf_contents(max_size=10)`         | Forwards to NumpyArray/string/bytestring                                |

### `regular_array_contents` naming collision

`regular_array_contents(max_size=5)` means "max group size" (the `size`
parameter of `RegularArray`), not "max total elements." This is a different
parameter with the same name. Renaming the top-level `max_size` to
`max_leaf_size` avoids confusion but does not resolve the collision within
`regular_array_contents`.

### Budget flow in `contents()`

```text
contents(max_size=M)
├── leaf → leaf_contents(max_size=min(M, max_length))
├── regular → recurse(max_size=M)        # full budget passed to child
├── list_offset → recurse(max_size=M)    # full budget passed to child
├── list → recurse(max_size=M)           # full budget passed to child
├── record → content_lists(max_total_size=M)  # budget distributed
└── union → content_lists(max_total_size=M)   # budget distributed
```

For single-child wrappers (regular, list_offset, list), the full `max_size`
budget is passed to the child. The wrapper itself adds no budget accounting.

### Budget tracking: `_leaf_size()`

```python
def _leaf_size(c: Content) -> int:
    return sum(len(leaf) for leaf in iter_leaf_contents(c))
```

Only counts leaf elements. Does not count offsets, indices, tags, or mask
buffers.

## Buffer Overhead per Content Type

### Types with no additional buffers

| Type         | Own buffers      | Notes                                              |
| ------------ | ---------------- | -------------------------------------------------- |
| NumpyArray   | data (n scalars) | Leaf; dtype varies                                 |
| EmptyArray   | none (0 scalars) | Leaf; no data                                      |
| RegularArray | none             | Logical grouping; `size` is metadata, not a buffer |
| RecordArray  | none             | Composition; children have all data                |

### Types with index/offset buffers

| Type            | Own buffers                       | Scalar count | Child scalars   |
| --------------- | --------------------------------- | ------------ | --------------- |
| ListOffsetArray | offsets: int64[n+1]               | n + 1        | all child       |
| ListArray       | starts: int64[n], stops: int64[n] | 2n           | all child       |
| UnionArray      | tags: int8[n], index: int64[n]    | 2n           | sum of children |

String and bytestring content are `ListOffsetArray` wrapping
`NumpyArray(uint8)`. Their overhead is covered by the `ListOffsetArray` row
above.

## Top-Down Budget Enforcement

The key change is that `contents()` must **deduct buffer overhead** before
passing the remaining budget to children.

### Single-child wrappers

```text
contents(max_size=M) → list_offset
  overhead = max_length + 1     # offsets array
  child_budget = M - overhead
  child = recurse(max_size=child_budget)
```

But there's a **chicken-and-egg problem**: the overhead of `ListOffsetArray`
depends on `n` (the number of lists), which is drawn inside
`list_offset_array_contents()`. We don't know `n` until after the child is drawn
and the offsets are generated.

### Resolution options

#### Option A: Estimate overhead before drawing

Estimate the maximum possible overhead (assume `n = max_length`) and deduct it
from the budget before drawing the child. This is conservative — the child gets
less budget than strictly necessary.

```python
# In contents(), before recursing into list_offset:
estimated_overhead = max_length + 1  # worst case for offsets
child_budget = max(max_size - estimated_overhead, 0)
child = draw(recurse(max_size=child_budget))
```

**Pros:** Simple; budget never exceeded. **Cons:** Over-reserves; small
`max_size` values may leave nothing for the child.

#### Option B: Post-hoc validation with `assume()`

Draw the child, draw the wrapper, then validate the total size. Reject and
redraw if over budget.

```python
child = draw(recurse(max_size=max_size))
result = draw(list_offset_array_contents(child, max_length=max_length))
assume(total_size(result) <= max_size)
```

**Pros:** No over-reservation. **Cons:** Rejection-based; can be slow with
Hypothesis if many draws are rejected. Hypothesis may also raise
`HealthCheck.too_slow`.

#### Option C: Move budget into wrapper strategies

Change the wrapper strategies to draw their own children instead of receiving
them. Each wrapper strategy would accept a child factory (callable returning a
strategy) and a total budget, draw its own parameters, deduct overhead, then
draw the child with the remaining budget.

```python
@st.composite
def list_offset_array_contents(
    draw: st.DrawFn,
    st_content: Callable[..., st.SearchStrategy[Content]],
    *,
    max_total_size: int = 10,
    max_length: int = 5,
) -> Content:
    n = draw(st.integers(min_value=0, max_value=max_length))
    overhead = n + 1  # offsets
    child_budget = max(max_total_size - overhead, 0)
    child = draw(st_content(max_size=child_budget))
    # Build ListOffsetArray from child and n
```

In the current design, the child is drawn **before** the wrapper — `contents()`
draws the child via `recurse()` and passes it as a concrete `Content` to the
wrapper strategy. The wrapper then slices the child into lists but never
controls the child's size. Option C reverses this: the wrapper draws its own
parameters first, deducts overhead, then draws the child with the remaining
budget.

**Pros:** Exact budget; each wrapper owns its budget logic. **Cons:** Changes
the wrapper strategy signatures from accepting `Content` to accepting a child
factory. Standalone use of wrapper strategies (outside `contents()`) becomes
less convenient — callers must wrap concrete children in `st.just()`.

#### Option D: Keep budget logic in `contents()`, draw parameters first

Keep the wrapper strategies simple (receive concrete child), but restructure
`contents()` to draw wrapper parameters before the child:

1. Choose wrapper type
2. Draw wrapper parameters (n lists, n tags, etc.) — deduct overhead
3. Draw child with remaining budget
4. Pass child to wrapper strategy

```python
case 'list_offset':
    n = draw(st.integers(min_value=0, max_value=max_length))
    overhead = n + 1
    child_budget = max(max_size - overhead, 0)
    child = draw(recurse(max_size=child_budget))
    result = draw(
        st_ak.contents.list_offset_array_contents(child, max_length=n)
    )
```

The wrapper strategy receives a concrete child and `max_length=n` (already
decided), so it does not redraw `n`.

**Pros:** Exact budget; wrapper strategies keep their current signatures.
**Cons:** Budget logic and wrapper parameter knowledge are duplicated —
`contents()` must know each wrapper's overhead formula. The wrapper strategies
still draw `n` internally, so `contents()` would need to constrain `max_length`
to the already-chosen `n` to avoid a second draw.

#### Option E: Hybrid — use current flow, cap child budget conservatively

Keep the current structure where the child is drawn first, but use a reasonable
estimate for overhead. For `ListOffsetArray`, the overhead is at most
`max_length + 1`. For `ListArray`, at most `2 * max_length`. For `UnionArray`,
at most `2 * total_union_length` (where `total_union_length` is
`sum(len(c) for c in children)`, i.e., the length of the tags/index arrays).

Use `max_length` defaults (5 for list types) to estimate overhead:

```python
case 'list_offset':
    ml = max_length if max_length is not None else 5
    child_budget = max(max_size - (ml + 1), 0)
    child = draw(recurse(max_size=child_budget))
    result = draw(list_offset_array_contents(child, max_length=ml))
```

**Pros:** Minimal refactoring; works with current structure. **Cons:** Slightly
conservative; overhead estimate depends on `max_length` defaults matching the
wrapper strategy defaults.

### Multi-child wrappers (record, union)

`content_lists()` already distributes a budget across children. Currently it
tracks `_leaf_size()`. This would change to track `_content_size()` which
includes index/offset buffers. For `UnionArray`, the tags/index overhead (`2n`)
should also be deducted from the budget.

## New `_content_size()` Function

```python
def _content_size(c: Content) -> int:
    '''Count total scalars stored in a content tree.'''
    match c:
        case NumpyArray():
            return len(c.data)
        case EmptyArray():
            return 0
        case RegularArray():
            return 1 + _content_size(c.content)
        case RecordArray():
            n_fields = len(c.fields) if c.fields is not None else 0
            return n_fields + sum(_content_size(child) for child in c.contents)
        case ListOffsetArray():
            return len(c.offsets.data) + _content_size(c.content)
        case ListArray():
            return len(c.starts.data) + len(c.stops.data) + _content_size(c.content)
        case UnionArray():
            return (len(c.tags.data) + len(c.index.data)
                    + sum(_content_size(child) for child in c.contents))
```

## Open Questions

1. **Should `max_size` in `regular_array_contents` be renamed?** It currently
   means "max group size," not "max elements." This is a separate naming issue
   from the `max_size`/`max_leaf_size` split.

2. **Which resolution option (A–E) for top-down enforcement?** Option D (draw
   wrapper params first) is most precise but requires the most refactoring.
   Option E (conservative estimate) is simplest.

3. **Should option types also have their mask buffers counted?** Option types
   are not yet implemented (see
   `.design/research/2026-03-27-option-types-research.md`). `IndexedOptionArray`
   has an int64 index, `ByteMaskedArray` has an int8 mask, `BitMaskedArray` has
   a uint8 mask. These should be counted for consistency when implemented.
